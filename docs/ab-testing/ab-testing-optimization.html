Traditional stopping rules use p-values to control error rates — specifically, the probability of falsely claiming a winner (Type I error). This is statistically sound, but it answers the wrong question.

Instead, we need to ask: what stopping rule maximizes our long-term outcomes?

What Drives Stopping Criteria?

In order to determine optimal stopping rules, we’re going to simulate 100,000 experiments like the one below:

The pre-experiment conversion rate. Sets the variance of your test statistic and determines how much room for improvement exists.
Total users in the experiment and their control/treatment split. More traffic means faster evidence. Uneven splits trade speed for risk protection.
The population exposed to the winner after the decision is made. More post-test traffic means more value at stake — justifying a more aggressive stopping rule.
The realistic lift you expect comes from a distribution of possible effects — not a single number. Experiments have an equal chance of being good or bad.
Control: Each simulation is a different realization of the same baseline, but different noise. This is the randomness inherent in experimentation.

Treatment: Each simulation of Treatment has a different true effect. Sometimes harmful, sometimes beneficial, often marginal. Even when we collect test data, we never know the exact truth.

Optimal Stopping Algorithm

For each of 100,000 simulated experiments and each significance level α ∈ [0.01, 0.50]:

  1. Burn-in: No stopping decisions before min_test_days = 7
  2. Daily check: Stop when either condition’s Pr(Best) exceeds 1 - α
  3. Fallback: If neither condition breaches by the final day:
    • Select Best: roll out whichever condition is currently ahead
    • Retain Control: keep control regardless
  4. Rollout: Deploy the winner to remaining test users and post-test traffic
  5. Evaluate: Pick the α with the highest average long-term outcome

As a baseline, we also simulate a Fixed Sample test — no early stopping, decision made only at max_test_days = 20 — using the Retain Control fallback.

Your Optimal Stopping Rule Is Unique to You — The right threshold shifts with your fallback policy, post-test traffic, and baseline rate. Simulate your context, find your curve, pick the peak.

import simulationengine as se

# ── Experiment Assumptions ─────────────────────────────────────────
CONTROL_MEAN       = 0.30      # pre-experiment conversion rate
MAX_TEST_N         = 100_000   # total users in the experiment
SPLIT              = 0.50      # treatment allocation (50/50)
MIN_TEST_DAYS      = 7         # mandatory burn-in before stopping decisions
NUMBER_TEST_DAYS   = 20        # maximum experiment duration
POST_TEST_N        = 500_000   # users exposed to winner once MAX_TEST_N is reached
NUMBER_SIMULATIONS = 100_000   # simulations to run
SEED               = 42
ITC_MEAN           = 100       # index-to-control mean
ITC_SE             = 1         # index-to-control standard error

df = se.create_simulation_df(
    number_test_days=NUMBER_TEST_DAYS,
    number_simulations=NUMBER_SIMULATIONS,
    seed=SEED,
    condition_parameters=[
        {
            "mean": CONTROL_MEAN,
            "trials": int(MAX_TEST_N * SPLIT),
        },
    ],
)

random_effect_parameters_0 = {
    "itc_mean": ITC_MEAN,
    "itc_se": ITC_SE,
    "trials": int(MAX_TEST_N * SPLIT),
}

df = se.add_random_effect_condition(df, random_effect_parameters_0)
df = se.add_cumulative_columns(df)
df = se.compute_cumulative_std_rates(df)
df = se.compute_delta_stats(df)
df = se.compute_test_stats(df)
df = se.calculate_remaining_test_potential(df)
df = se.calculate_launch_potential(df, post_test_n=POST_TEST_N)
df = se.calculate_total_potential(df)

# ── KDE density + survival function for ITC mini-chart ───────────
from scipy.stats import gaussian_kde as _kde, norm as _norm
_kde_fn   = _kde(itc_values)
_x_dense  = np.linspace(float(itc_values.min()), float(itc_values.max()), 400)
_y_dense  = _kde_fn(_x_dense)
_itc_mean = float(itc_values.mean())
_itc_std  = float(itc_values.std())
_sf       = [round(float(_norm.sf(x, _itc_mean, _itc_std)) * 100, 2) for x in _x_dense]
_split    = _x_dense <= 100
itc_density = {
    "xL": _x_dense[_split].tolist(),  "yL": _y_dense[_split].tolist(),
    "sfL": [_sf[i] for i in range(len(_x_dense)) if _split[i]],
    "xR": _x_dense[~_split].tolist(), "yR": _y_dense[~_split].tolist(),
    "sfR": [_sf[i] for i in range(len(_x_dense)) if not _split[i]],
    "mean": _itc_mean,
    "std": _itc_std,

# Optimal Stopping Algorithm

For each of 100,000 simulated experiments and each significance level α ∈ [0.01, 0.50]:

1. **Burn-in:** No stopping decisions before `min_test_days = 7`
2. **Daily check:** Stop when either condition's Pr(Best) exceeds `1 - α`
3. **Fallback:** If neither condition breaches by the final day:
   - _Select Best_: roll out whichever condition is currently ahead
   - _Retain Control_: keep control regardless
4. **Rollout:** Deploy the winner to remaining test users and post-test traffic
5. **Evaluate:** Pick the α with the highest average long-term outcome

As a baseline, we also simulate a **Fixed Sample** test — no early stopping, decision made only at `max_test_days = 20` — using the _Retain Control_ fallback.

::: {.callout-note}
**P-Hacking is Good for Business** — Stopping early when you see a signal is optimal when the goal is value, not academic rigor. The optimal threshold here is α = 0.12 — stricter just means defaulting to control unnecessarily.

**Your Optimal Stopping Rule Is Unique to You** — The right threshold shifts with your fallback policy, post-test traffic, and baseline rate. Simulate your context, find your curve, pick the peak.

**Inconclusive Tests Have a Winner Too** — When neither condition clears the bar, picking the current leader over retaining control adds 35 events per experiment. Multiply that across a year of testing.
